# TAVA: **Open-Vocabulary Action Recognition via Text Retrieval Augmentation and Visual Attribute Association**

## Introduction

We propose TAVA, a novel OVAR framework that utilizes visual attributes as an explicit bridge to capture both distinctions and associations between base and novel actions. Specifically, TAVA augments action category representations through retrieval over training data, thereby expanding the effective class space, and employs large language models to obtain corresponding visual attribute information for base categories. To further facilitate generalization, we introduce an Augmented Category Constraint to improve recognition of novel actions and an Attribute Relation Constraint to enhance the learning of fine-grained visual distinctions across categories. Extensive experiments on HMDB51, UCF101, SSv2, and Kinetics-400/600 demonstrate that TAVA consistently outperforms state-of-the-art methods under diverse open-vocabulary settings.

## Data Preparation

### 1) Video frames / videos

This code expects **frame/video roots** via `DATA.PATH_PREFIX`.

CSV files in `label_db/` and `zs_label_db/` contain rows formatted as:

`<relative_video_path>,<label_id>`

and `DATA.PATH_LABEL_SEPARATOR` should match the delimiter (typically `,`).

Recommended layout (edit paths in commands):

- K400 videos: `data/k400/`
- HMDB51 videos: `data/hmdb51/videos/`
- UCF101 videos: `data/ucf101/videos/`
- K600 videos: (see your local path) for example `data/k600/videos/`

### 2) Label splits and mappings

#### Base-to-Novel (B2N)

Label splits are stored under `zs_label_db/`:

- `zs_label_db/B2N_k400/` (K400 base training split; contains `train_*.csv`, `base_val.csv`, `novel_val.csv`, and `train_rephrased.json` / `test_rephrased.json`)
- `zs_label_db/B2N_hmdb/` (HMDB51 base/novel splits; contains `val.csv` for base and `test.csv` for novel)
- `zs_label_db/B2N_ucf101/` (UCF101 base/novel splits; contains `val.csv` for base and `test.csv` for novel)

#### Cross-Dataset

Use the “full” / “open-set split” resources under `zs_label_db/` and the mappings under `label_db/` or `zs_label_db/`:

- HMDB51 full: `zs_label_db/hmdb_full/test.csv` + `zs_label_db/hmdb-index2cls.json`
- UCF101 full: `zs_label_db/ucf101_full/test.csv` + `zs_label_db/ucf101-index2cls.json`
- K600 open-set split example (split1): `zs_label_db/k600_split1/test.csv` + `zs_label_db/k600_split1/k600-openset-index2cls.json`

### 3) Attribute files (used by B2N training when enabled)

Attribute dictionaries are provided in `attribute_files/`:

- `final_visual_attributes_k400.json`
- `final_visual_attributes_hmdb.json`
- `final_visual_attributes_ucf.json`

When your training config enables the attribute-based training branch, you must set:

- `DATA.ACTION_ATTRIBUTE_MAPPING_FILE`
- `DATA.PROXY_LABEL_MAPPING_FILE`

## Training
To reproduce the paper settings, use the provided shell scripts (they wrap `tools/run_net.py` with the correct `--cfg` / `--opts` arguments and may run SWA/weight-averaging automatically).

- Base-to-Novel (B2N, HMDB51): `bash script/training/temporal_b16/train_clip_B2N_hmdb.sh`
- Cross-dataset (K400 backbone): `bash script/training/temporal_b16/train_clip.sh`

Other datasets have parallel scripts in the same folders. Please edit `CUDA_VISIBLE_DEVICES` / `NUM_GPUS` / `ROOT` / `CKPT` inside the scripts to match your machine (the paper uses 4 GPUs).

## Implementation Details

We adopt CLIP with a **ViT-B/16** backbone and use **FROSTER** as the baseline for comparison. All experiments are implemented in PyTorch and trained on **4 NVIDIA RTX 4090 GPUs**.

We use **Adam** with initial learning rate `3.33e-6`, decayed to `3.33e-8` using **cosine scheduling**. The model is trained for:

- **Base-to-Novel setting**: **12 epochs**
- **Cross-dataset setting**: **22 epochs**

Data sampling follows the paper:

- **Training**: randomly sample **8 frames** per video, resize to **224 x 224**, and use **random cropping** + **horizontal flipping**
- **Testing**: sample **three 8-frame clips** per video, apply **center cropping**, and **average clip-level predictions**

Run the following scripts directly (the scripts already fix the corresponding `--cfg/--opts`; you only need to edit `ROOT/CKPT`-related paths and `CUDA_VISIBLE_DEVICES/NUM_GPUS` inside the scripts):

- HMDB51 ：`bash script/training/temporal_b16/train_clip_B2N_hmdb.sh`
- UCF101 ：`bash script/training/temporal_b16/train_clip_B2N_ucf.sh`
- K400 ：`bash script/training/temporal_b16/train_clip_B2N_k400.sh`
- SSv2 ：`bash script/training/temporal_b16/train_clip_B2N_ssv2.sh`

Training (K400 backbone):
- `bash script/training/temporal_b16/train_clip.sh`

Evaluation (starting from the trained checkpoint):
-  To evaluate using the trained checkpoint, run the corresponding scripts listed under `## Evaluation` in `script/testing/temporal_test_b16/` (they set `LOAD_CKPT_FILE`, `DATA.PATH_*`, and `MODEL.NUM_CLASSES`).
